# SanWich Workers 完整版授權機制規劃

> 文件狀態：規劃完成，尚未實作
>
> 適用專案：SanWich 目前桌面版
>
> 新的對外名稱：免費功能、完整版（Full）
> 本文件取代 `SUPPORTER_PLAN.md` 作為後續授權系統的主要規格。

## 1. 決策摘要

- SanWich 維持同一套安裝程式，不拆成免費版與付費版兩個安裝包。
- 第一次安裝自動開啟 14 天完整版試用，但試用不占用正式授權的裝置名額。
- 使用者付款後取得一組唯一 Key，在 SanWich 內輸入 Key 啟用完整版。
- 一組正式 Key 最多同時啟用 2 台裝置。
- Cloudflare Worker 是授權判斷入口，Cloudflare D1 保存 Key、裝置與操作紀錄。
- 正式啟用必須連線；啟用後每 30 天重新驗證一次，另有 14 天離線寬限。
- 免費功能不因授權伺服器故障、Key 到期或驗證失敗而失效。
- Beta Key 與正式付費 Key 必須分批管理，可設定到期日並一次停用整批 Beta Key。
- 對外 UI 與使用說明不再使用 `Supporter`。Key 成功啟用後只顯示「完整版」或 `Full`。
- 現有離線 Key 無法被遠端停用；要套用新機制，使用者必須安裝支援 Workers 授權的新版本。

## 4. 使用者付款與 Key 配發流程

### 4.1 Beta 與初期正式營運：人工配發

目前可沿用既有付款網站，不需要先重做購物網站。流程如下：

1. 使用者在現有網站完成付款。
2. 付款平台寄通知信給管理者。
3. 管理者核對付款金額與訂單資料。
4. 管理者進入授權後台建立一組新的完整版 Key。
5. 設定 Key 類型、到期日、最多裝置數與備註。
6. 管理者將 Key 寄給使用者。
7. 使用者在 SanWich 的「完整版啟用」區貼上 Key。
8. SanWich 連線到 Worker；啟用成功後顯示「完整版」。

人工配發適合 Beta 與初期少量訂單。缺點是寄信、查單與補發都需要人工處理。

### 4.2 後續可選：付款後自動寄發

只有在付款平台提供可靠的 Webhook 或 API 時才自動化：

1. 付款成功事件送到 Worker。
2. Worker 驗證 Webhook 簽章與訂單狀態。
3. 系統產生唯一 Key，建立 D1 授權紀錄。
4. 系統寄出 Key 或顯示領取頁面。
5. 重複通知不得重複發 Key，必須以訂單編號做冪等處理。

未確認付款平台 API 前，不把自動發信列為第一階段必要功能。

## 5. Trial、正式啟用與驗證計算

### 5.1 第一次安裝的 14 天試用

- 第一次安裝自動開啟的是「完整版試用」，不是正式完整版授權。
- 不需要輸入 Key。
- 不占用正式 Key 的 2 台裝置名額。
- 預設沿用現有本機 Trial 記錄與多重錨點，不強迫第一次啟動連網。
- 刪除單一授權檔不應重設 Trial；仍沿用 `%APPDATA%`、`%LOCALAPPDATA%` 與登錄檔的復原策略。
- 本機 Trial 只能提高重設成本，無法達到伺服器等級的防濫用。

### 5.2 正式啟用何時計數

只有符合以下全部條件時，才新增 1 個裝置名額：

1. 使用者輸入有效的 Beta Key 或完整版 Key。
2. SanWich 成功連到 Worker。
3. Worker 確認 Key 狀態可用且尚未超過裝置上限。
4. Worker 成功建立新的裝置紀錄並回傳簽章授權 Token。

以下行為不增加裝置數：

- 第一次安裝進入 14 天完整版試用。
- 同一台裝置每 30 天重新驗證。
- 同一台裝置重新開啟程式。
- IP、電腦名稱、Windows 更新或顯示卡改變。
- 在同一套 Windows 安裝中更換主機板，但本機裝置私鑰仍可讀取。

## 6. 一組 Key 最多兩台裝置

### 6.1 認定原則

- 上限是「同時存在的有效裝置紀錄」，不是累計安裝次數。
- 每組正式 Key 預設 `max_devices = 2`。
- 同一位使用者可用在桌機與筆電各 1 台。
- 第 3 台裝置啟用時，Worker 必須拒絕並引導使用者停用舊裝置。

### 6.2 裝置身分

不得只用主機板、CPU、硬碟或 MAC 位址作為唯一裝置身分。建議流程：

1. SanWich 第一次正式啟用時，在本機產生一組裝置公私鑰。
2. 私鑰使用 Windows DPAPI 保護，不以明文寫入 `license.json`。
3. Worker 只保存裝置公開金鑰與必要的裝置顯示資訊。
4. 每次驗證時由本機簽署 Worker 提供的 nonce，證明仍持有該裝置私鑰。
5. 硬體資訊只做柔性風險判斷，不做單一封鎖依據。

這樣更換主機板、顯示卡或硬碟時，只要 Windows 與受保護私鑰仍存在，就不應直接失效。

### 6.3 重灌、壞機與換機

- 使用者可在仍能開機的舊裝置按「停用此裝置」。
- 舊電腦壞掉或已重灌時，可到授權管理頁面停用舊裝置。
- 第一次正式啟用時提供一次性顯示的授權復原碼；管理頁面要求 Key 加復原碼，不能只憑 Key 就允許停用裝置。
- 使用者遺失復原碼時，以付款訂單或購買證明人工處理。
- 每組 Key 每 90 天允許 1 次自助替換裝置。
- 超過自助替換頻率後進入人工確認，不直接永久封鎖。
- 所有停用、重設與人工處理都寫入不可任意修改的操作紀錄。

此規則的目的，是讓正常換機有路可走，同時避免同一組 Key 在多人之間反覆輪替。

## 7. 每 30 天重新驗證與離線寬限

### 7.1 建議時間

- 正常重新驗證週期：30 天。
- 離線寬限：14 天。
- Worker 回傳的簽章 Token 最長有效 45 天。
- 到達第 30 天時主動驗證；若伺服器或網路暫時不可用，可在剩餘 14 天內繼續使用完整版。
- 超過 Token 有效期限仍無法完成驗證時，只關閉完整版功能，免費功能繼續使用。

### 7.2 Key 被停用時

- 成功連線且 Worker 回覆 `revoked`、`expired` 或 `suspended` 時，立即停止新的完整版授權。
- 已在離線狀態的使用者，最晚可用到現有 Token 到期。
- 不能宣稱能讓完全離線、未更新或已修改程式的用戶端立即失效。

### 7.3 Beta 指定日期截止

- Beta Token 的 `valid_until` 不得晚於 Beta Key 的 `expires_at`。
- 即使一般 Token 可用 45 天，Beta 到期日較早時仍以 Beta 到期日為準。
- Beta 結束前應在程式內顯示到期提醒與正式購買方式。
- 時間到後，Beta 完整版功能結束，免費功能保留。

## 8. Beta 50 組 Key 的管理方式

### 8.1 發放

- 每位測試者取得不同 Key，不共用同一組萬用 Key。
- 全部標記同一批次，例如 `beta_2026_01`。
- 每組 Key 設定 `license_type = beta`、`max_devices = 2` 與明確 `expires_at`。
- 發放時明確說明這是限時 Beta 測試授權，不承諾永久完整版。

### 8.2 Beta 結束

管理後台必須支援依批次執行：

- 全部停用。
- 全部延長到期日。
- 選定部分測試者轉正式授權。
- 匯出使用與驗證摘要。

Beta Key 到期或停用後，不應直接變成正式付費 Key。正式購買後另發新的完整版 Key，讓 Beta 與付費紀錄清楚分開。

### 8.3 舊版限制

目前已發出的舊式離線 Key 無法被 Worker 遠端撤銷。若 Beta 測試已使用舊版本，必須：

1. 發布支援 Workers 授權的新版本。
2. 要求測試者更新。
3. 以新的 Beta Key 取代舊式離線 Key。

舊程式若不更新，仍可能繼續接受舊式離線 Key。這是遷移限制，不得對外宣稱已能完全遠端停用。

## 9. Cloudflare 架構

### 9.1 元件

| 元件 | 用途 |
| --- | --- |
| Cloudflare Worker | 啟用、重新驗證、停用、裝置管理與管理者 API |
| Cloudflare D1 | Key、裝置、批次、訂單參照與操作紀錄 |
| Cloudflare Pages 或 Worker 靜態頁面 | 使用者裝置管理頁與管理者後台 |
| Workers Secrets | 管理密鑰、伺服器簽章私鑰、Webhook Secret |

授權伺服器的私鑰、管理密碼與付款 Webhook Secret 絕不可放進公開 GitHub、桌面程式或發佈 ZIP。

### 9.2 建議 API

| 方法與路徑 | 用途 |
| --- | --- |
| `POST /v1/license/activate` | 使用 Key 啟用目前裝置 |
| `POST /v1/license/validate` | 每 30 天重新驗證與更新 Token |
| `POST /v1/license/deactivate` | 從仍可使用的裝置停用自己 |
| `POST /v1/license/recover` | 以 Key 與復原碼管理已失聯裝置 |
| `GET /v1/license/devices` | 列出遮罩後的有效裝置 |
| `POST /v1/admin/keys` | 建立 Key |
| `PATCH /v1/admin/keys/:id` | 停用、延長、暫停或修改上限 |
| `POST /v1/admin/batches/:id/revoke` | 整批停用 Beta Key |
| `POST /v1/admin/devices/:id/reset` | 人工釋放裝置名額 |

管理者 API 必須有獨立身分驗證、限流與操作紀錄，不能只靠一個難猜網址保護。

## 10. D1 資料模型

### 10.1 `license_keys`

- `id`
- `key_hash`
- `key_prefix`
- `license_type`：`beta`、`full`
- `status`：`active`、`suspended`、`revoked`、`expired`
- `batch_id`
- `max_devices`
- `expires_at`
- `created_at`
- `updated_at`
- `purchase_reference`
- `customer_email`：可選，僅在付款與補發確實需要時保存
- `note`

資料庫原則上只保存 Key 的高強度雜湊或伺服器端 HMAC，不保存可直接登入的完整明文 Key。完整 Key 只在建立時顯示一次；遺失時以撤銷舊 Key、補發新 Key 處理。

### 10.2 `devices`

- `id`
- `license_id`
- `device_public_key`
- `device_name`
- `os_summary`
- `status`
- `first_activated_at`
- `last_validated_at`
- `deactivated_at`
- `replacement_count`
- `replacement_window_started_at`

只保存授權管理必要資訊，不上傳字幕、音訊、影片、API Key、Prompt、個人規則或專案內容。

### 10.3 `audit_log`

- `id`
- `actor_type`：`user`、`admin`、`system`
- `actor_id`
- `action`
- `license_id`
- `device_id`
- `reason`
- `created_at`
- `request_id`

### 10.4 `orders`（第二階段可選）

只有付款自動化時才建立，用來保存訂單編號、付款狀態與對應的 Key。不得保存完整信用卡資料。

## 11. Key 格式與安全規則

- Key 必須由密碼學安全亂數產生，不能使用連號或可猜測資料。
- 建議新的對外前綴使用 `SWF`，例如 `SWF-XXXX-XXXX-XXXX-XXXX`；Beta 可使用 `SWB`。
- 資料庫不保存完整明文 Key。
- Worker 必須限制作答頻率，防止大量猜 Key。
- 啟用與驗證回應使用伺服器私鑰簽章；桌面程式只內建公開金鑰。
- 所有請求使用 HTTPS。
- 管理者操作使用獨立登入與多因素驗證。
- 使用者管理頁不能只輸入 Key 就允許停用裝置，必須再驗證復原碼或購買證明。
- 日誌不得記錄完整 Key；最多顯示前後遮罩字元。

SanWich 是公開原始碼程式，熟悉程式的人仍可能修改用戶端略過檢查。新機制的目標是管理正常付費使用與阻止一般 Key 分享，不宣稱能絕對防破解。

## 12. 使用者與管理者介面

### 12.1 SanWich 設定頁

建議顯示：

- 目前狀態：基本功能可用／完整版試用／完整版。
- 試用剩餘天數。
- 完整版 Key 輸入欄。
- 「啟用完整版」按鈕。
- 目前裝置名稱。
- 下次驗證日期。
- 「停用此裝置」按鈕。
- 授權異常時的離線寬限期限。
- 購買完整版連結。

Key 成功啟用後，狀態只顯示「完整版」，不顯示 Free、Trial 或 Supporter。

### 12.2 使用者裝置管理頁

- 輸入 Key 與復原碼。
- 顯示最多 2 台裝置及最後驗證日期。
- 裝置名稱必須遮罩，不顯示不必要的硬體序號。
- 可停用已壞掉或無法存取的裝置。
- 顯示 90 天自助替換限制與下次可替換日期。
- 復原碼遺失時提供購買證明人工處理方式。

### 12.3 管理者後台

- 建立單組或批次 Key。
- Key 類型：Beta、完整版。
- 狀態：有效、暫停、撤銷、到期。
- 設定到期日與最多裝置數。
- 查看裝置、首次啟用與最後驗證時間。
- 單獨停用 Key。
- 依批次停用全部 Beta Key。
- 釋放裝置名額。
- 記錄付款參照與客服備註。
- 查看完整操作紀錄。
- 匯出不含完整 Key 的管理報表。

## 13. 失敗與降級規則

| 情況 | 應有行為 |
| --- | --- |
| Worker 暫時無法連線 | 使用尚未到期的本機 Token，顯示重試資訊 |
| 30 天驗證日到但無網路 | 進入最多 14 天寬限 |
| Token 到期仍無法驗證 | 關閉完整版功能，免費功能照常可用 |
| Key 被管理者撤銷 | 下次成功驗證時關閉完整版功能 |
| 第 3 台裝置要求啟用 | 拒絕並引導停用舊裝置 |
| 本機授權資料損壞 | 嘗試復原；失敗時回到免費功能，不鎖整個程式 |
| 使用者系統時間倒退 | 不延長 Token；保留最後成功伺服器時間並要求重新驗證 |
| 管理後台誤操作 | 透過操作紀錄確認，允許重新啟用但不刪除歷史紀錄 |

## 14. 費用與規模

- 50 組 Beta Key 的請求與資料量極小。
- 初期可使用 Cloudflare Workers、D1 與 Pages 免費額度。
- 不使用自訂網域時，可直接使用 Cloudflare 提供的服務網址。
- 若使用自訂網域，主要固定成本是網域年費。
- 需要較高可用性或超過免費額度時，再升級 Workers 付費方案。
- 真正的主要成本是初次開發、測試、付款查核與換機客服，不是每 30 天驗證的流量。

## 15. 實作階段

### Phase 0：鎖定名稱與產品規則

- 對外統一使用「免費功能／完整版（Full）」。
- 確認完整版價格、Beta 截止日與每組 Key 最多 2 台。
- 決定現有舊 Key 是換發、保留或停止接受。

### Phase 1：Workers 與 D1 基礎

- 建立 D1 Schema 與 migration。
- 建立 Key 產生、啟用、驗證與撤銷 API。
- 建立伺服器簽章金鑰與 Secrets 管理。
- 加入限流、Key 遮罩與操作紀錄。

### Phase 2：桌面程式授權模組

- 保留 `has_feature(feature_name)` 對其他功能的既有介面。
- 將正式完整版判斷改為伺服器簽章 Token。
- 建立 DPAPI 保護的裝置私鑰。
- 實作 30 天驗證與 14 天寬限。
- 保留免費功能的零網路依賴。
- 更新所有對外文字為「完整版／Full」。

### Phase 3：管理者後台

- 建立 Key、批次 Key 與 Beta 批次。
- 單組與整批停用。
- 裝置名額與人工重設。
- 管理者身分驗證與操作紀錄。

### Phase 4：使用者換機管理

- 裝置清單。
- 復原碼。
- 壞機停用。
- 90 天自助替換限制。
- 購買證明人工復原流程。

### Phase 5：Beta 與遷移驗證

- 建立測試用 Beta 批次與到期日。
- 使用不同 Windows 裝置進行兩台／第三台測試。
- 測試重灌、換主機板、斷網、時間倒退與 Worker 故障。
- 確認舊版本與舊 Key 的處理方式後才正式發放新 Key。

### Phase 6：付款自動化（可選）

- 確認現有付款網站的 Webhook／API 能力。
- 驗證付款通知簽章。
- 自動產生與寄發 Key。
- 訂單冪等與退款撤銷流程。

## 16. 測試清單

- 第一次安裝進入 14 天完整版試用，不建立正式裝置名額。
- Trial 到期只關閉完整版功能。
- 有效 Key 可啟用第 1 台與第 2 台。
- 第 3 台被拒絕。
- 同一台重新驗證不增加名額。
- 更換電腦名稱、IP、顯示卡不產生新裝置。
- 同一 Windows 安裝更換主機板後仍能驗證。
- 重灌後可透過復原流程釋放舊名額。
- 90 天內重複換機被引導到人工處理。
- Worker 中斷時免費功能正常，完整版依 Token 與寬限運作。
- Beta 到期日不被 45 天 Token 延長。
- 管理者可停用單組 Key。
- 管理者可一次停用指定 Beta 批次的全部 Key。
- 被撤銷 Key 在下次驗證後失效。
- 本機 Token、Key 或授權檔被竄改時不會誤開完整版。
- 系統時間倒退不會延長完整版期限。
- 發佈 ZIP 不包含 Worker Secret、簽章私鑰、管理密碼或正式 Key。
- UI、README、設定頁與提示彈窗不再出現 Supporter。

## 17. 完成標準

Workers 授權機制只有在以下項目全部完成後，才能視為可正式發放：

- Key 由 Worker／D1 管理，不再只靠本機靜態驗證。
- 每組 Key 的兩台裝置上限經過實機測試。
- 主機板更換與 Windows 重灌有可驗證的處理流程。
- Beta 批次停用已實際測試。
- 30 天驗證與 14 天寬限已測試。
- 免費功能在伺服器故障時仍完整可用。
- 管理者後台有身分驗證、限流與操作紀錄。
- 使用者可停用壞機裝置，管理者可人工復原。
- 舊 Key 遷移方式已決定並對外說明。
- 所有對外名稱已改為「完整版／Full」。
- 隱私與發佈掃描確認沒有伺服器 Secret 或完整 Key 外洩。

## 18. 尚未決定、實作前必須確認

- 完整版是永久授權、年度授權，或未來兩者並存。
- 正式售價與退款後的 Key 處理規則。
- 現有付款網站是否支援 Webhook。
- 現有舊式 Key 是否全部換發。
- Beta 的明確結束日期。
- 使用者裝置管理頁使用獨立網域或 Cloudflare 預設網址。
- 14 天完整版試用已確定並實作。
