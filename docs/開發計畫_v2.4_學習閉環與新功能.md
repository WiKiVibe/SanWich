# SanWich v2.4 之後：學習閉環與新功能開發計畫

> 規劃日期：2026-07-14
>
> 基線版本：v2.3.2
>
> 狀態：僅完成規劃，尚未開始實作。需使用者確認後才進入開發。

## 一、開發原則

- 學習來源必須是使用者的明確行為，不把「AI 有照規則輸出」誤當成「使用者認為它是對的」。
- 所有學習資料預設只存在本機；音訊、字幕修改與規則不建立雲端帳號，也不自動同步。
- 每一條規則都要能追溯來源、查看證據、停用、降級、刪除與還原。
- 規則分成專案、系列／節目、領域、全域四層，避免某集受訪者姓名污染所有工作。
- Free 核心字幕流程永久可用；學習閉環、規則管理與重度省時功能維持 Supporter 定位。
- v2.x 繼續使用目前核心與 CustomTkinter；PySide6 只先做原型，不進行一次性全面重寫。
- 近期維持使用者自填 API Key，不內建共用 Key、不架中繼伺服器。
- BAT 輕量安裝包維持主要發佈形式；EXE 為次要項目，不阻擋功能版發佈。

## 二、學習閉環的完整資料流

1. Breeze 產生原始字幕。
2. AI 校對產生候選修改，並記錄哪些字幕實際完成校對。
3. 使用者在快速對照或字幕編輯器做出明確決定：
   - 接受 AI 修改。
   - 還原 Breeze 原文。
   - 手動改成第三種文字。
   - 略過，不提供學習訊號。
4. 系統把行為寫成不可變的回饋事件，不立即偷偷改規則。
5. 規則候選引擎彙整重複證據，判斷作用範圍與信心。
6. 達到門檻時顯示候選清單，由使用者確認加入、改範圍、忽略或永久拒絕。
7. 下一次 AI 校對只注入符合目前專案／系列／領域的有效規則。
8. 校對完成後再次收集人工確認，形成可持續循環。

## 三、資料結構規劃

### 1. 人工回饋事件

- 建議位置：`%APPDATA%\SanWich\learning\review_feedback.jsonl`。
- 每筆事件至少包含：
  - `event_id`、時間、SanWich 版本。
  - 專案／系列／領域識別。
  - 輸入檔案指紋，不保存完整媒體內容。
  - 字幕時間碼與原始 Breeze 文字。
  - AI 候選文字、使用者最後文字。
  - 行為：`accept_ai`、`restore_original`、`manual_edit`、`skip`。
  - 當時套用的規則 ID 與 AI 供應商／模型。

### 2. 個人化規則 schema v2

- 延續 `personal_rules.json`，加入：
  - `scope_type`：project／series／domain／global。
  - `scope_id`：專案或系列識別。
  - `state`：candidate／active／frozen／rejected。
  - `human_accept_count`、`human_reject_count`、`manual_confirm_count`。
  - `evidence_event_ids`、`last_confirmed_at`、`confidence`。
  - `created_by`：manual／suggested／migrated。
- 現有 schema v1 規則遷移為 `active + domain`，保留原採納統計，不破壞既有資料。

### 3. 專案與系列設定

- 新增本機 `project_profiles.json`。
- 可保存節目名、受訪者、固定用語、領域與預設 Prompt 範圍。
- 專案資料不自動從檔名推斷為永久規則；第一次由使用者選擇或建立。

## 四、人工行為如何轉成學習訊號

- 「接受 AI」：對該次修改給一筆正向人工證據。
- 「還原原文」：對該次 AI 修改及相關規則給一筆負向證據。
- 「手動修改」：以 Breeze／AI → 最終文字建立最強的候選證據。
- 「略過」：只記操作完成度，不增加正負分數。
- 同一替換在相同系列或領域被人工確認至少 2 次，才主動建議升級為規則。
- 使用者手動按「直接加入規則」時可以立即啟用，但仍保留來源事件。
- 規則連續被還原 2 次時先暫停並提示檢查，不直接永久刪除。
- 全域規則的升級門檻高於專案／系列規則，避免姓名與單集術語誤套。

## 五、版本與階段

### v2.4.0：回饋事件地基

- 把快速對照的接受、還原、手動修改、略過接到事件紀錄。
- 把 SRT 編輯器匯出差異接到同一種事件格式。
- 將編輯歷史移到 `%APPDATA%` 持久路徑，並提供舊 `logs/srt_edit_history.jsonl` 一次性遷移。
- 建立事件檢視與清除入口，但暫不自動升級規則。
- 驗收：四種行為都能正確記錄；關閉重開後仍存在；不含 API Key 或完整媒體。

### v2.4.1：候選規則與人工確認

- 將事件彙整成候選規則。
- 新增專案／系列／領域／全域範圍選擇。
- 候選視窗顯示出現次數、正反證據、原句上下文與來源集數。
- 支援加入、延後、永久忽略、合併與改範圍。
- 驗收：單次偶發修改不會自動污染規則庫；重複確認能穩定提出候選。

### v2.4.2：閉環評分與規則健康度

- AI 呼叫前依範圍、人工信心、最近使用時間選規則。
- AI 完成後不再以文字比對直接增加「人工採納」，只記模型遵循度。
- 規則管理器分開顯示「模型有遵守」與「人類確認正確」。
- 被連續還原的規則自動暫停；提供一鍵復原。
- 驗收：錯規則能被人工行為壓低與暫停，正確規則能逐步提高優先度。

### v2.5.0：剪輯師工作流補強

- 主畫面加入「直接開啟外部 SRT」，不必先跑一次轉寫。
- 語者分離增加「SRT 顯示語者」選項，預設仍關閉。
- 建立專案／系列快速選擇，讓規則作用範圍可見。
- 驗收：外部 SRT 可直接編輯、播放、檢查與匯出；語者標籤不破壞原時間碼。

### v2.5.1：長音訊與首次下載體驗

- 顯示已完成時間、估計剩餘時間與目前處理階段。
- Breeze 首次下載加入實際進度、失敗重試、磁碟空間與中斷說明。
- CPU 模式依音訊長度提供較保守的時間預期。
- 驗收：下載失敗不會留下看似完整的模型；重新啟動可以清楚重試。

### v2.6.0：Supporter 新功能

- 領域 Prompt 模板：訪談、新聞、教學、科技與自訂模板。
- 自訂詞庫：姓名、品牌、地名、術語，可指定專案／系列範圍。
- Prompt 模板與詞庫統一走 `has_feature()`，並可查看實際送入 AI 的內容摘要。
- 驗收：模板、詞庫、個人規則不互相覆蓋；Free 模式不在背景偷偷套用。

### v2.6.x：效能實驗，通過評測才正式加入

- Silero VAD：評估長訪談靜音切分、速度與幻覺率。
- Breeze CTranslate2／faster-whisper 後端：比較速度、VRAM、台灣口音準確率與時間碼品質。
- 每項實驗都使用同一批真實訪談素材，保留基準數據與可回退開關。

### v3 原型：PySide6

- 只先製作主視窗＋大型字幕時間軸原型。
- 比較視窗拖動、2,000 組字幕、波形縮放、DPI、記憶體與打包體積。
- 原型明顯優於 v2 才建立 v3 分支，依序移植主流程、編輯器、設定與規則庫。
- macOS、雲端同步、中繼伺服器、團隊規則庫等議題延後到 v3 基礎穩定後評估。

## 六、測試策略

- 單元測試：事件 schema、字幕對齊、候選聚合、作用範圍、信心與規則升降級。
- 整合測試：Breeze 原文 → AI 修改 → 人工操作 → 事件 → 候選 → 下一次 Prompt。
- GUI 測試：快速對照四種操作、外部 SRT、2,000 組字幕、錯誤時間碼跳轉。
- 授權測試：Free、Trial、Supporter 三種模式都驗證 UI 與背景行為一致。
- 遷移測試：舊設定、舊規則、舊編輯歷史在更新後不遺失、不重複匯入。
- 隱私測試：分享 ZIP 與事件檔不得含 API Key、媒體內容、完整本機絕對路徑或未經同意的雲端傳輸。

## 七、每一階段的完成定義

- 程式碼、資料遷移、UI、測試與使用說明必須同時完成。
- 新功能必須有關閉方式與資料回退方式。
- 自動測試通過後，仍需用真實訪談 SRT 做人工驗收。
- 分享包必須重新做版本、必要檔案、敏感資料、乾淨安裝與主視窗啟動檢查。
- 未達成驗收條件的功能只能標記為實驗，不併入穩定版。
